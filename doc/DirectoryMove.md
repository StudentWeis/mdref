# mdref 目录移动实现说明

本文档说明 `mv` 支持目录移动后的整体思路，重点解释为什么要先规划、再执行，以及链接更新是如何完成的。

## 一、目标

目录移动要解决的不只是文件系统中的路径变化，还要同时保证 Markdown 中的相对链接仍然有效。

这项能力需要覆盖两类影响：

- 目录外部文件指向该目录内部文件的链接
- 目录内部 Markdown 文件指向目录外部文件的链接

同时，操作过程中还要尽量保证原子性。也就是说，如果移动或改写中途失败，项目不能落在“目录已经挪走一半、引用只更新了一部分”的状态。

## 二、核心思路

整体流程可以概括为四步：

1. 校验源路径和目标路径是否合法
2. 递归建立目录内所有旧路径到新路径的映射
3. 基于路径映射规划所有需要改写的 Markdown 链接
4. 执行目录移动，并在失败时回滚

这里最关键的一点是：**先规划，后执行**。

如果一上来就直接移动目录，后面再慢慢找链接并更新，一旦中途失败，恢复成本会很高。先把需要改什么都算清楚，执行阶段就可以更接近事务式操作。

## 三、为什么目录内相互引用通常不需要改

如果目录整体移动，目录内部文件之间的相对位置并没有变化。

例如：

- `docs/guide.md` 链接 `docs/nested/topic.md`
- 对应相对路径是 `nested/topic.md`

当整个 `docs` 被移动到 `archive/docs` 后：

- `archive/docs/guide.md`
- `archive/docs/nested/topic.md`

两者之间的相对关系没有变，所以 `nested/topic.md` 仍然成立。

因此，目录移动时真正需要重新计算的，主要是“跨目录边界”的链接。

## 四、路径映射

实现中会递归遍历源目录，为每个目录项建立路径映射：

- `old/path/a.md -> new/path/a.md`
- `old/path/nested/b.md -> new/path/nested/b.md`

这样做有两个作用：

- 外部文件引用目录内部文件时，可以通过旧目标路径快速找到新目标路径
- 如果某个被引用的文件本身也在被移动目录内，引用文件的新位置也可以一起推导出来

换句话说，目录移动不是单纯地“处理一个目录”，而是把它转成“一组路径迁移关系”。

## 五、链接改写策略

### 1. 外部引用目录内部文件

先用现有的 `find_references` 找出所有指向目标目录的链接。

这一步的目录扫描会尊重 `.gitignore` 等标准忽略规则，因此被忽略目录中的 Markdown 文件不会参与这类引用改写。

对每条引用：

1. 解析出它原来指向的绝对路径
2. 在路径映射表中找到新的目标路径
3. 以“引用文件的新位置”为基准重新计算相对路径

如果引用文件本身不在被移动目录内，它的新位置就是原路径；如果它本身也在被移动目录内，就需要先通过路径映射推导出它的新位置。

### 2. 目录内部引用外部文件

目录中的 Markdown 文件会整体换位置，因此这些文件内部指向目录外部的链接会失效，需要重算。

处理方式是：

1. 递归扫描目录内所有未被忽略的 Markdown 文件
2. 找出它们包含的本地链接
3. 如果链接目标不在被移动目录内，就按移动后的文件位置重新计算相对路径

### 3. 目录内部互相引用

这类链接通常不改，因为相对位置保持不变。

## 六、执行和回滚

真正执行阶段不是逐文件复制，而是直接对目录做一次 `rename`。

这样做有三个好处：

- 更符合“移动目录”的语义
- 性能和实现复杂度都更好
- 失败时可以整体移回原位置

回滚机制会记录两类信息：

- 哪些文件在改写前做过内容快照
- 这次移动是“复制型”还是“重命名型”

如果目录移动后的链接改写失败，就会：

1. 把目录从目标位置重命名回源位置
2. 恢复所有已修改文件的原始内容

这样能尽量保证操作对外表现为原子化。

## 七、一个简单例子

### 移动前

```text
project/
├── index.md
├── shared/
│   └── faq.md
└── docs/
    ├── guide.md
    └── nested/
        └── topic.md
```

`index.md` 内容：

```md
[Guide](docs/guide.md)
[Topic](docs/nested/topic.md)
```

`docs/guide.md` 内容：

```md
[FAQ](../shared/faq.md)
[Topic](nested/topic.md)
```

执行命令：

```sh
mdref mv docs archive
```

### 移动后

```text
project/
├── index.md
├── shared/
│   └── faq.md
└── archive/
    ├── guide.md
    └── nested/
        └── topic.md
```

`index.md` 会被更新为：

```md
[Guide](archive/guide.md)
[Topic](archive/nested/topic.md)
```

`archive/guide.md` 会被更新为：

```md
[FAQ](../shared/faq.md)
[Topic](nested/topic.md)
```

这里有两个关键点：

- `index.md` 是目录外部文件，所以它指向 `docs/...` 的链接必须改成 `archive/...`
- `guide.md` 到 `topic.md` 的链接不需要改，因为它们在移动前后仍然保持相对位置不变

## 八、边界条件

当前实现已经处理的情况：

- 如果目标路径已经存在，则直接报错
- 如果目标目录是源目录自身或其子目录，则直接拒绝执行
- 如果开启 `dry-run`，只输出计划中的移动和改写，不真正修改文件
- 外部 URL（`https://`、`mailto:` 等）不会被改写
- 纯锚点链接（`#section`）不会被改写
- 带锚点的文件链接（`file.md#section`）会正确保留锚点部分
- 断链（指向不存在文件的链接）会被跳过，不会导致报错
- URL 编码的路径（如 `%20` 表示空格）能正确解析
- Link Reference Definition 语法（`[label]: url`）能正确识别和改写

## 九、总结

目录移动的本质不是“批量搬文件”，而是“批量重建路径关系”。

实现上先把所有路径变化和链接变化规划出来，再执行目录重命名和内容改写，最后用事务式回滚兜底。这样既复用了现有的查找与相对路径计算能力，也把目录移动控制在了比较清晰、可验证的边界内。

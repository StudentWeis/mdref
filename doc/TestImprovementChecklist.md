# 测试体系改进清单

本文档基于当前 `mdref` 测试体系的现状评估，整理出后续可执行的改进事项，便于按优先级逐步推进。

## 目标

- 降低测试重复率，明确测试分层职责。
- 补齐薄弱模块的测试覆盖。
- 提升测试可维护性、稳定性与跨平台兼容性。
- 建立更清晰的测试质量基线。

## P2：提高可读性与维护性

### 7. 统一测试命名规范

- [ ] 统一使用 `test_<对象>_<场景>_<预期>` 命名模式。
- [ ] 重命名信息量不足的测试。
- [ ] 避免继续新增 `basic`、`normal`、`simple` 这类泛化命名。

**优先重命名示例：**

- [ ] `test_cli_find_basic`
- [ ] `test_cli_mv_basic`
- [ ] `test_cli_rename_basic`
- [ ] `test_find_links_basic`
- [ ] `test_find_references_basic`
- [ ] `test_rename_basic`

**验收标准：**

- [ ] 只看测试名即可大致判断场景与断言目标。

### 9. 统一 fixture 构造模式

- [ ] 复用已有 benchmark fixture 的思路，整理常见测试目录结构模板。
- [ ] 为“单文件引用”“多文件引用”“目录迁移”“Unicode 路径”建立标准 fixture 构造函数。
- [ ] 减少每个测试文件里手写目录树的样板代码。

**验收标准：**

- [ ] 测试关注点更多落在行为断言，而不是文件系统搭建细节。

---

## P3：稳定性与平台兼容性

### 10. 处理平台相关测试问题

- [ ] 评估 `tests/bench_script_tests.rs` 的 Unix 限制是否可缩小到具体测试用例。
- [ ] 梳理 `std::os::unix::*` 依赖，避免不必要的平台绑定。
- [ ] 明确哪些测试必须是 Unix-only，哪些逻辑应补跨平台验证。
- [ ] 如计划支持 Windows，补充 Windows 平台的最小验证路径。

**验收标准：**

- [ ] 平台相关限制有清晰说明。
- [ ] 非平台专属逻辑尽量保持跨平台可跑。

### 11. 补充文件系统边界场景

- [ ] 增加只读文件或权限不足场景测试。
- [ ] 增加符号链接链路相关测试。
- [ ] 评估大小写不敏感文件系统相关风险。
- [ ] 增加 CRLF / LF 行尾差异测试。
- [ ] 评估 BOM、非法 UTF-8 等输入的处理策略。

**验收标准：**

- [ ] 常见文件系统差异不会在发布后才暴露。

---

## P4：质量基线与流程建设

### 12. 增加覆盖率度量

- [ ] 引入 `cargo llvm-cov` 或等价方案。
- [ ] 输出覆盖率报告并记录基线。
- [ ] 重点观察 `src/commands`、`src/core/rename.rs`、错误分支覆盖率。

**验收标准：**

- [ ] 能快速识别测试薄弱区域。
- [ ] 后续重构前后可量化对比覆盖率变化。

### 13. 区分测试执行层次

- [ ] 区分快速测试与较慢测试。
- [ ] 评估 benchmark 相关测试是否需要单独分组。
- [ ] 在文档中说明常用执行方式。
- [ ] 将 `script/precheck.sh` 作为默认质量检查入口写入测试文档。

**建议补充到文档中的常用命令方向：**

- [ ] 仅跑核心测试
- [ ] 仅跑某个测试文件
- [ ] 跑完整 precheck

**验收标准：**

- [ ] 开发者能按场景快速选择合适的测试命令。

### 14. 建立增量改进节奏

- [ ] 每次 PR 至少完成一个“去重复”或“补空白覆盖”动作。
- [ ] 测试重构与功能开发同步推进，避免长期堆积。
- [ ] 新增功能默认先补测试，再补实现。

**验收标准：**

- [ ] 测试体系改进可以小步持续推进，而不是一次性大重写。

## 建议执行顺序

### 第三阶段

- [ ] 推广 `rstest`
- [ ] 统一命名规范
- [ ] 统一 fixture 模式

### 第四阶段

- [ ] 增加覆盖率度量
- [ ] 处理平台兼容性
- [ ] 完善测试执行文档

---

## 备注

- 当前测试体系的主要问题不是“完全缺少测试”，而是“重复较多、层次混杂、薄弱点不均衡”。
- 建议优先做结构优化，再继续横向扩充测试数量。
- 如需进一步拆解，可继续产出按文件维度的详细改造计划。

# mdref Architecture

本文档介绍 `mdref` 当前版本的整体架构，重点说明模块分层、调用链路、核心数据流，以及与文件系统相关的设计约束。

## 1. 项目定位

`mdref` 是一个面向 Markdown 引用关系的命令行工具，当前主要提供三类能力：

- `find`：查找某个 Markdown 文件被哪些 Markdown 文件引用，同时列出该文件内部包含的链接。
- `mv`：移动文件，并同步更新全项目中指向该文件的 Markdown 链接。
- `rename`：重命名文件，本质上是 `mv` 的特化形式。

从代码组织上看，这个项目同时暴露了两层接口：

- CLI 层：面向终端用户，由 `src/main.rs` 和 `src/commands/` 组成。
- Library 层：面向复用和测试，由 `src/lib.rs` 暴露的 `core` 能力组成。

这意味着 `mdref` 的可执行程序只是一个很薄的适配层，真正的业务逻辑都集中在库层。

## 2. 顶层模块结构

当前源码可以概括为下面几层：

```text
src/
├── main.rs           # CLI 入口
├── lib.rs            # 对外暴露库 API
├── error.rs          # 统一错误类型
├── commands/         # 命令层：参数到库调用的适配
│   ├── mod.rs
│   ├── find.rs
│   ├── mv.rs
│   └── rename.rs
└── core/             # 核心能力：查找、迁移、引用表示
    ├── mod.rs
    ├── find.rs
    ├── mv.rs
    └── reference.rs
```

各层职责如下：

- `main.rs`：解析命令行参数，调用命令分发函数，并将错误输出到标准错误。
- `commands/`：把 CLI 参数转换为具体的库调用，负责控制台输出，但尽量不承载业务逻辑。
- `core/`：实现真正的 Markdown 链接扫描、引用匹配、路径重写和文件移动逻辑。
- `error.rs`：定义统一的 `MdrefError` 和 `Result<T>`，作为命令层和核心层共享的错误抽象。

## 3. 执行流

### 3.1 命令入口

程序从 `src/main.rs` 启动：

1. 使用 `clap` 解析命令行参数。
2. 将解析结果映射为 `commands::Commands` 枚举。
3. 调用 `commands::handle_command` 分发到对应命令实现。
4. 若返回错误，则统一打印并以非零退出码结束进程。

这个入口保持得很薄，意味着新增命令时只需要：

- 在 `commands::Commands` 中扩展一个子命令。
- 在 `commands::handle_command` 中增加分支。
- 在 `commands/` 和 `core/` 中补充对应实现。

### 3.2 `find` 命令流程

`find` 命令的行为可以拆成两个步骤：

1. 调用 `find_references(file_path, root_path)`，扫描根目录下所有 Markdown 文件，找出引用目标文件的记录。
2. 调用 `find_links(file_path)`，解析目标文件自身包含的所有 Markdown 链接。

命令层只负责输出分段标题和结果列表；扫描与匹配逻辑全部位于 `core::find`。

### 3.3 `mv` / `rename` 命令流程

`rename` 并没有独立的核心实现，它会先把旧文件名和新文件名拼成路径，再直接调用 `mv_file`。因此，迁移逻辑只有一套，差别只在于目标路径的计算方式。

`mv_file` 的执行顺序如下：

1. 确保目标路径的父目录存在。
2. 调用 `find_references` 找出所有引用旧文件的 Markdown 位置。
3. 先将原文件复制到新路径。
4. 遍历引用结果，更新所有外部引用文件中的链接路径。
5. 调用 `find_links(new_file_path)` 解析新文件内部的链接。
6. 遍历这些内部链接，按新位置重写相对路径。
7. 删除旧文件。

这种顺序体现了当前实现的两个设计意图：

- 先复制再删原文件，避免更新过程中出错导致源文件丢失。
- 先修复“别人指向我”的链接，再修复“我指向别人”的链接。

## 4. 核心数据模型

项目目前只有一个显式的数据结构：`Reference`。

```rust
pub struct Reference {
    pub path: PathBuf,
    pub line: usize,
    pub column: usize,
    pub link_text: String,
}
```

这个结构统一表示一次 Markdown 链接命中，既可以表示：

- 某个文件中“引用了目标文件”的一条记录。
- 某个文件内部“包含了某个链接”的一条记录。

字段含义如下：

- `path`：包含链接的 Markdown 文件路径。
- `line`：链接所在行号，1-based。
- `column`：链接所在列号，1-based。
- `link_text`：`[text](link)` 中的 `link` 部分。

这个模型很轻量，适合打印和批量处理，但也意味着它没有保存更多上下文，例如：

- 链接文本 `text`。
- 原始整行内容。
- 链接的语义类型（图片、文档、锚点、外部 URL）。

因此当前实现对“Markdown AST 级别”的复杂处理支持较弱，整体仍然是文本扫描驱动。

## 5. 查找子系统

查找能力集中在 `src/core/find.rs`，可分成四个阶段。

### 5.1 文件遍历

`find_references` 使用 `walkdir` 遍历 `root_dir` 下的文件，并只保留扩展名为 `.md` 的条目。

这里的遍历通过 `rayon::par_bridge()` 转成并行流，因此每个 Markdown 文件的读取和扫描可以并发执行。这个设计适合项目 README 中强调的“快速扫描目录”的目标。

### 5.2 链接识别

链接识别依赖一个正则表达式：

```text
\[([^\]]*)\]\(([^)]+)\)
```

也就是识别标准的 `[text](link)` 结构。当前实现因此具备以下特点：

- 优点：实现简单、执行快、易于并行化。
- 代价：不处理更复杂的 Markdown 语法变体，例如嵌套括号、引用式链接、带 title 的链接等。

### 5.3 路径解析与匹配

`process_link` 负责判断某个链接是否指向目标路径，其逻辑是：

1. 若没有指定目标路径，说明当前只是收集链接，直接接受。
2. 若目标是文件，先做一次文件名快速过滤，减少不必要的路径解析。
3. 调用 `resolve_link` 按“相对于当前 Markdown 文件所在目录”的方式解析链接。
4. 对解析出的路径做 `canonicalize`，与目标路径做规范化后的比较。

对于目录目标，当前逻辑使用 `starts_with(target)` 判断链接是否位于该目录下。这和 README 里的“支持按目录搜索”目标一致，不过当前文档和测试对目录场景的覆盖仍然偏少。

### 5.4 结果收集

扫描命中后会构造 `Reference`，记录文件路径、行列和原始链接文本。`find_links` 和 `find_references` 都复用同一个底层函数 `process_md_file`，差别只在于是否传入目标路径。

这使得“收集全部链接”和“筛选命中目标的链接”共享一套扫描逻辑，降低了维护成本。

## 6. 迁移子系统

迁移能力位于 `src/core/mv.rs`，核心问题不是“移动文件”，而是“保证迁移后所有相对链接仍然正确”。

### 6.1 外部引用更新

对于引用旧文件的每一个 `Reference`，系统会：

1. 取出引用文件路径 `r.path`。
2. 计算从该引用文件到新文件的相对路径。
3. 在对应行内执行字符串替换，将 `](old_link)` 替换为 `](new_link)`。

其中相对路径计算依赖 `pathdiff::diff_paths`，这是当前迁移逻辑的关键基础设施。

### 6.2 被移动文件内部链接更新

文件移动后，文件内部原本的相对链接可能失效。例如：

- 原文件 `docs/a.md` 链接到同目录下的 `b.md`。
- 若 `a.md` 被移动到 `archive/a.md`，则链接需要从 `b.md` 改写为 `../docs/b.md` 或其他新的相对路径。

为了解决这个问题，`mv_file` 会在新文件生成后再次调用 `find_links(new_file_path)`，遍历该文件中的全部链接，并按下面规则重写：

- 如果链接原本指向旧文件自身，则改写成新文件文件名。
- 否则，计算从新文件位置到原链接目标文件的相对路径。

这部分逻辑由 `update_link` 完成。

### 6.3 写回策略

当前写回实现是“按引用逐个读写文件”：

1. 读入整个 Markdown 文件。
2. 找到目标行。
3. 在该行上做字符串替换。
4. 将整个文件重新写回磁盘。

这个实现直接、容易理解，但也带来几个工程上的后果：

- 同一个文件如果包含多处引用，可能被重复读写多次。
- 字符串替换依赖 `](link)` 片段完全匹配，不是基于语法树或精确 span 的编辑。
- 整个迁移流程不是事务性的，中途失败时可能出现“部分文件已更新”的状态。

这几个点并不影响当前工具的基本可用性，但它们是后续演进时最值得优先优化的地方。

## 7. 错误处理

项目统一使用 `MdrefError` 表达错误，当前包含三类：

- `Io`：文件读取、写入、复制、删除、规范化等 IO 相关错误。
- `Path`：路径缺失父目录、文件名等逻辑错误。
- `InvalidLine`：按 `Reference.line` 回写时，目标行号超出文件范围。

命令层不细分错误类型，只做统一打印；因此错误分类更多是服务于库层 API 和测试，而不是用户交互层面的精细提示。

## 8. 测试与性能验证

### 8.1 测试结构

项目通过 `tests/` 下的集成测试覆盖两类核心能力：

- `lib_find_tests.rs`：验证链接提取、引用查找、行列信息和若干边界条件。
- `lib_mv_tests.rs`：验证文件移动、引用改写、内部链接修复和目录创建行为。

这些测试的特点是：

- 主要围绕公开库 API 编写，而不是直接测试内部私有函数。
- 会动态创建临时目录和 Markdown 文件，覆盖真实文件系统交互。

这种测试方式和 `mdref` 的架构匹配得较好，因为项目的核心行为本质上就是“读取磁盘、计算路径、写回磁盘”。

### 8.2 Benchmark

`benches/benchmark.rs` 使用 `criterion` 基于生成的 `mock_data/` 数据集测试 `find_references` 性能。Mock 数据生成器会递归创建目录、文件和相互引用，用来模拟大规模扫描场景。

这也印证了当前架构的一个重点优化方向：扫描阶段追求吞吐量，尤其依赖：

- `walkdir` 进行目录遍历。
- `rayon` 进行并行处理。
- 正则和路径规范化进行快速过滤。

## 9. 当前架构的设计特点

结合实现细节，可以把 `mdref` 的架构风格总结为以下几点：

- 薄 CLI、厚 Library：命令层尽量只做参数适配和输出。
- 文本驱动：核心处理基于正则和字符串替换，而不是 Markdown AST。
- 文件系统优先：路径规范化、相对路径计算和磁盘写回是设计中心。
- 查找与迁移复用同一套引用模型：`Reference` 贯穿搜索、展示和修改流程。
- 并行优化集中在扫描阶段：迁移阶段仍然是串行、逐条写回。

这种架构很适合当前项目规模，因为它实现成本低、可读性高、性能也足够好。但随着功能扩展，下面几个方向可能会逐渐成为瓶颈：

- Markdown 解析精度不足。
- 多次重复写回同一文件。
- 缺少预览模式和事务性保障。
- 目录级移动和更复杂链接类型的覆盖不够完整。

## 10. 后续演进建议

如果未来继续扩展，可以优先考虑下面几个方向：

1. 在 `core` 内引入“扫描结果聚合层”，按文件批量更新，避免重复读写。
2. 将正则解析替换为更稳健的 Markdown 解析方案，至少对链接 span 做更精确定位。
3. 为 `mv_file` 增加预览模式或 dry-run 模式，先展示改动，再执行写回。
4. 为迁移流程增加回滚或阶段性校验，减少部分写入导致的不一致状态。
5. 补充目录目标、空格路径、复杂链接语法等测试场景。

## 11. 一句话总结

`mdref` 当前采用的是一种非常实用主义的 Rust 架构：用轻量 CLI 驱动一个以文件系统和相对路径计算为核心的库层，通过并行扫描快速发现引用，再用直接的文本改写完成迁移。
